# 高风险热点清单（优先级从高到低）

### A 级（立刻投入精力）

1. **Net / VirtIO / PCI × Logic Error**
   - 典型风险：feature 协商/队列长度/描述符边界/多队列状态机判断错误 → OOB、信息泄露、DoS。
   - 热点路径：`hw/virtio/*`, `hw/net/*`, `net/*`, `hw/pci/*`, `drivers/vhost/*`
   - 关注函数/模式：`virtio_queue_pop/push`、`virtio_*_set_features`、`virtqueue_map_desc`、length/feature 位判断、环形队列边界检查。
2. **Memory / MMU / Paging × Misuse of API**
   - 典型风险：地址空间/FlatView/MemoryRegion API 调用顺序或标志错误；TLB 刷新/权限位漏同步 → 非法访问、越权读写、崩溃。
   - 热点路径：`softmmu/memory.c`, `include/exec/memory.h`, `accel/*`, `target/*/mmu*`
   - 关注函数/模式：`memory_region_add_subregion`、`address_space_rw`、TLB flush/权限映射/页属性更新的原子性。
3. **TCG / Translation Engine × Hardware Emu Error**
   - 典型风险：指令译码/访存语义不一致（size/对齐/符号扩展） → 客体可控崩溃、静默错误、潜在逃逸前置条件。
   - 热点路径：`tcg/*`, `target/*/translate*`, `target/*/op_helper*`, `include/exec/*`
   - 关注函数/模式：`MO_SIZE/*`, `tcg_temp_new*`、访存 helper、译码表边界、未覆盖/错误 default 分支。
4. **Migration / Snapshot × Logic Error**
   - 典型风险：VMState 版本化/字段可选与缺省值/来源与目的端兼容处理错误 → 迁移中断、状态损坏、恢复后行为异常。
   - 热点路径：`migration/*`, `qapi/migration*`, `vmstate_*`
   - 关注函数/模式：VMState 字段读写次序、长度/标志位校验、跨版本兼容、失败路径释放/回滚一致性。
5. **Block I/O Layer × Resource Leak / Bounds**
   - 典型风险：镜像格式（qcow2/raw/vmdk）元数据边界检查、图锁（bdrv_graph）顺序、I/O 长度/对齐 → OOB、挂起、资源泄漏。
   - 热点路径：`block/*`, `hw/block/*`, `qemu-img/*`
   - 关注函数/模式：`bdrv_*` 调用顺序、length/offset 边界、镜像元数据解码与路径回收。

------

### B 级（重点跟进）

1. **Interrupts / Timers × Race Condition**
   - 典型风险：icount/定时器/中断注入与设备状态机并发访问 → 死锁/丢中断/重复触发。
   - 路径：`hw/timer/*`, `qemu-timer*`, `apic/gic/*`
2. **Core Infra / Logging × Misuse of API**
   - 典型风险：`error_setg/errp` 使用不当、日志路径改变程序流（隐藏错误/误判恢复） → 静默失败/错误边界被穿越。
   - 路径：`util/*`, `qapi/*`, `monitor/*`, `trace-events*`
3. **Net / VirtIO / PCI × Resource Leak**
   - 典型风险：失败路径缺释放、DMA/unmap 漏掉，导致可被 VM 驱动重复触发 → Host 资源耗尽 DoS。
4. **Memory / MMU / Paging × Race Condition**
   - 典型风险：页属性变更/映射重建与并发访存冲突；缺乏原子/屏障 → 崩溃/脏读。
5. **TCG × Logic Error**
   - 典型风险：断言/边界条件过窄，译码或 Helper 前置条件判断错误 → 可控崩溃。

> 注：上述排序依据你安全子集的交叉图强度 + 经典易错点的先验知识综合而来，能直接指导挖掘与验证。

------

# 结构化观察与结论

1. **数据路径 > 控制路径的安全风险**
   - `Hardware Access` 与 `Resource Lifecycle` 在 Net/VirtIO/PCI、Memory/MMU、Block 层的交叉格往往更“重”——因为**长度字段/对齐/DMA/环形队列**这些都是典型“可控数据面”。
   - 结论：**优先从“可被 Guest 控制的数据面”入口做审计与 fuzz**（Virtqueue、PCI config、MMIO 寄存器、迁移流）。
2. **TCG 是“系统性风险放大器”**
   - 指令翻译细节错误会把微小的不一致放大为**可被特制指令序列稳定复现**的崩溃或越界；
   - 结论：**TCG 翻译/访存语义需要独立的规则集**（尺寸/对齐/符号扩展/默认分支/未覆盖指令）。
3. **API 误用是“稳定的漏洞来源”**
   - Memory API（FlatView/MemoryRegion）和错误处理（`error_setg`/`ERRP_GUARD`）的误用，常见且隐蔽；
   - 结论：**建立“调用次序 + 失败路径回收”的静态检查清单**可以明显降低误用类问题。
4. **迁移/快照的“跨版本兼容”是高发风险点**
   - VMState 的可选字段/版本化容易出现“写多读少/默认值不当”，复现难、危害大；
   - 结论：**对迁移流构建 schema 验证 + fuzz**（长度字段畸形、字段缺失/重复）。
5. **Block I/O 的边界/图锁问题是 DoS 的温床**
   - 镜像元数据解析 + bdrv 图操作一旦边界或锁顺序出错，容易导致挂起/泄漏；
   - 结论：**I/O 路径单元化（小输入片段）+ 图锁顺序审计 + 失败回滚检查**。

# 直接可用的落地动作（按热点逐项给方法）

### ① Net/VirtIO/PCI × Logic Error（A1 顶级热点）

- **静态规则**：
  - `virtio_queue_*` 返回值未校验；`desc.len` 与 `iov/sg` 长度不匹配；多队列/packed vs split 模式分支遗漏；feature 位未 gate 住路径。
- **Fuzz 策略**：
  - 以 `virtio-net/blk/scsi/9p` 为目标；变异 vring 描述符长度/标志/索引；模拟设备 reset/热插拔/feature 变更；开启 ASAN/UBSAN。
- **验证清单**：
  - “环边界 + feature 组合 + 失败路径释放”三联测试；
  - 迁移中途变更 feature/队列尺寸的恢复逻辑。

### ② Memory/MMU/Paging × Misuse of API（A2）

- **静态规则**：
  - `address_space_rw` 长度/对齐检查；`memory_region_add_subregion` 后忘了必要的 sync/flush；TLB 刷新顺序/缺失；访问权限位未同步。
- **Fuzz 策略**：
  - 建立 MMIO/页表畸形场景；对页属性变更时并发访存；迁移前后访问权限突变。

### ③ TCG × Hardware Emu Error（A3）

- **静态规则**：
  - `MO_*` 的尺寸与访存 helper 参数不一致；译码表 default 分支吞异常；符号扩展缺失；未对齐访存不处理。
- **Fuzz 策略**：
  - 指令序列生成器（随机/语义导向），重点覆盖访存/原子指令与边界尺寸组合。

### ④ Migration/Snapshot × Logic Error（A4）

- **静态规则**：
  - VMState 字段版本/标志位不一致、读写次序不同；失败路径缺回滚；跨版本默认值处理不对。
- **Fuzz 策略**：
  - 变异迁移流长度/字段顺序/缺失；跨版本迁移回放；中断注入 + 迁移并发。

### ⑤ Block I/O × Resource Leak / Bounds（A5）

- **静态规则**：
  - `bdrv_*` 调用顺序/错误分支缺 unref/close；图片元数据读取越界；写路径 length/offset 未校验。
- **Fuzz 策略**：
  - 针对 qcow2/raw/vmdk 元数据结构做 AFL++ 结构化变异；小片段 I/O + 并发 flush/discard。

# 关键结论（来自 6 张交叉图）

## 1) Functionality × RootCause

- **Control Path Logic × Logic Error**：最显著、分布最广。大量属于“条件/状态判断不当、断言/边界缺失”的通用缺陷。
- **Hardware Access × Logic Error**：紧随其后，集中在设备寄存器/队列/长度字段/对齐校验等数据面入口，安全相关性更强。
- **Resource Lifecycle × Resource Leak**：明显存在，失败路径与异常回滚不完整是高发模式。
- **Hardware Emu Error**：在 TCG 与设备模型（MMIO/指令语义）里有“簇状”出现，容易导致可复现崩溃或静默错误。

> 启示：优先检查“**数据面** + **控制分支** + **失败路径回收**”三类组合，命中面最广、产出效率最高。

------

## 2) Functionality × Module

- **Hardware Access** 在 **Net / VirtIO / PCI、Memory / MMU / Paging、Block I/O** 三大模块里最重——这三处都是客体可控输入（队列/描述符/页属性/镜像元数据）的集中入口。
- **Control Path Logic** 在 **Migration / Snapshot、TCG** 明显；迁移的版本化/兼容分支与 TCG 的翻译分支都是容易“漏一个 case、坏一大片”的地方。
- **Instrumentation/Logging** 占比较低，但和错误处理绑定时（`error_setg/ERRP_GUARD`）会转化为“静默失败”风险。

> 启示：对 **VirtIO/PCI、MMU/Memory、Block** 做“输入结构 + 边界 + 对齐 + 长度”规则化审计，对 **Migration、TCG** 做“分支完备性 + 版本化/译码表覆盖”审计。

------

## 3) RootCause × Module

- **Logic Error** 是几乎所有模块的“底噪”，但在 **Net/VirtIO/PCI、Migration、Block** 中尤其显著。
- **Misuse of API** 在 **Memory / MMU / Paging** 与 **Core Infra** 更集中（MemoryRegion/FlatView/TLB/errp 调用序和约束最易错）。
- **Hardware Emu Error** 在 **TCG** 与目标设备相关目录明显（译码/访存语义/默认分支/符号扩展）。
- **Race Condition** 在 **Interrupts/Timers**、**Migration** 里可见，计时/注入与设备状态并发是常见模式。
- **Resource Leak** 在 **Block** 与 **Net** 比较多，典型是失败路径遗漏释放或 dma/unmap 不对称。

> 启示：按模块定制规则最有效：
>  Memory 侧做 **API 使用顺序/权限同步/TLB 刷新**；
>  TCG 做 **译码/访存语义一致性**；
>  VirtIO/PCI 做 **环形队列/特性协商/边界**；
>  Block 做 **元数据边界/图锁顺序/rollback**。

------

## 4) Functionality × Consequence

- **Hardware Access → Security Impact** 相关性强（越界/OOB、UAF、信息泄露多来自数据面访问）。
- **Control Path Logic → Crash/Abort** 显著（断言/分支缺失导致快速崩溃）。
- **Resource Lifecycle → Hang/Deadlock / Security Impact** 均有（泄漏/双重释放导致耗尽或可利用）。

> 启示：一条“访问路径 + 分支校验 + 资源回收”的组合检查能覆盖三类后果。

------

## 5) RootCause × Consequence

- **Logic Error** 更常见地落在 **Crash/Abort**。
- **Resource Leak** 与 **Hang/Deadlock**/DoS 关联明显；当涉及 fd/memory/bdrv 等关键资源时也会落到 Security。
- **Misuse of API** 既可能导致 **Functional Incorrectness**（行为不符），也会在 Memory API 使用不当时转为 **Security Impact**。
- **Hardware Emu Error** 在 **Crash/Abort** 与 **Incorrectness** 之间分布较均匀。

> 启示：对“Security Impact”的贡献主要来自 **Hardware Access、Leak、API Misuse** 三类根因与 Memory/VirtIO 这些数据面模块的交叠。

------

## 6) Module × Consequence

- **TCG、Memory/MMU** 在 **Crash/Abort** 上权重高（翻译/页表错误容易直接崩）。
- **Net/VirtIO/PCI、Block** 的 **Security Impact** 占比更高（可控输入 + 底层数据面）。
- **Migration/Timers** 对 **Hang/Deadlock** 有贡献（长事务/并发时序问题）。

> 启示：安全研究的两条主线：
>  ① **Net/VirtIO/PCI、Block**：以“安全影响”为主，优先做 fuzz 与边界审计；
>  ② **TCG、Memory/MMU**：以“崩溃/一致性”为主，优先做语义规则与严谨单测。

------

# 趋势图里的共性变化（来自 4 张趋势图）

- **Consequence 趋势**：Crash/Abort 与 Security Impact 总体随“功能迭代周期”起伏——接近合入大特性或重构阶段有高峰；
- **Module 趋势**：Net/VirtIO/PCI、Memory/MMU、Migration 出现“波峰”，与特性开发/兼容性工作（迁移、PCI/virtio 扩展）节奏相关；
- **RootCause 趋势**：Logic Error 始终“稳高”，Misuse of API 与 Resource Leak 在特定模块迭代期会抬头（如 Memory API、bdrv 图操作调整期）；
- **Functionality 趋势**：Hardware Access 与 Control Path Logic 基本主导，Resource Lifecycle 在重构/重写后期出现抬头（收尾阶段暴露）。

> 启示：把**发布节奏/大特性窗口**叠加到趋势图上，可以更准确地在“高风险窗口期”投放审计/Fuzz 资源。

# 直接可落地的行动清单

1. **RulePack（静态规则）优先 5 条**
   - VirtIO 队列：`desc.len`/`iov` 长度一致性 + feature gate + 多队列/packed-split 分支完备；
   - Memory API：`address_space_rw` 长度/权限检查 + `memory_region_add_subregion` 后置同步/flush + TLB 刷新时序；
   - TCG：`MO_*` 尺寸与 helper 参数一致性 + 未覆盖译码分支告警 + 符号扩展/未对齐访存；
   - Migration：VMState 字段读写次序/默认值/跨版本兼容 + 失败路径回滚对称性；
   - Block/bdrv：元数据长度/offset 边界 + `bdrv_*` 调用顺序 + 失败路径正确 unref/close。
2. **Fuzz 目标与输入设计（按模块）**
   - VirtIO/PCI：变异 vring 描述符、feature、index、reset/热插拔；
   - Memory/MMU：页属性批量切换 + 并发访存 + 非对齐/越界映射；
   - TCG：指令序列生成器覆盖访存/原子组合；
   - Migration：流结构变异（长度/顺序/缺失）+ 跨版本回放；
   - Block：镜像元数据畸形（qcow2/raw/vmdk）+ 小片段并发 I/O。
3. **热点审计清单**（按目录切任务）
   - `hw/virtio/*`, `hw/net/*`, `net/*`, `hw/pci/*`
   - `softmmu/memory.c`, `include/exec/*`, `target/*/mmu*`
   - `tcg/*`, `target/*/translate*`, `target/*/op_helper*`
   - `migration/*`, `qapi/migration*`
   - `block/*`, `hw/block/*`, `qemu-img/*`
